Gemfiles

Gemfile에는 최소한 하나의 젬 소스가 필요합니다. 젬 소스에는 루비젬 서버가 URL형식으로 적혀있어야합니다. bundle init를 실행함으로써 rubygems.org 를 소스로한 기본값의 Gemfile을 만듭니다. 할 수만 있다면 https을 사용하여 rubygems.org 서버로의 연결이 SSL인증되도록 하세요. 
source 'https://rubygems.org'
필요한 젬들을 버전 번호와 함께 선언합니다. 의존성을 지원하기위한 버전 지정에는 루비젬에서 사용하는 것과 같은 문법을 사용합니다. 
gem 'nokogiri'
gem 'rails', '3.0.0.beta3'
gem 'rack',  '>=1.0'
gem 'thin',  '~>1.1'
>= 1.0같은 대부분의 버전 지정자는 자기 자신을 설명합니다. 지정자 ~>는 특별한 의미가있습니다. 예제로 설명하겠습니다. ~> 2.0.3는 >= 2.0.3 그리고 < 2.1와 같습니다. ~> 2.1는 >= 2.1 그리고 < 3.0와 같습니다. ~> 2.2.beta는 2.2.beta.12같은 프리릴리즈 버전과 같습니다.
루비젬의 버전 지정자
젬의 메인 파일이 젬의 이름과다르다면, require하는 방법을 지정할 필요가 있습니다. 
gem 'rspec', :require => 'spec'
gem 'sqlite3'
:require => false을 지정함으로써 번들러가 젬을 require하는것을 방지할 수있습니다. 하지만 여전히 인스톨과 의존성의 유지보수는 이루어 집니다. 
gem 'rspec', :require => false
gem 'sqlite3'
Gemfile안의 젬들을 require하려면, 어플리케이션에서 Bundler.require를 콜할 필요가 있습니다.
자세한 내용은: Bundler.require
깃 저장소들도 유효한 젬 소스입니다. 저장소가 하나이상의 유효한 젬을 가지고 있다면 말이죠. :tag나 :branch, :ref로 어느걸 체크아웃할지 정할 수 있습니다. 기본값은 master브랜치 입니다. 
gem 'nokogiri', :git => 'git://github.com/tenderlove/nokogiri.git', :branch => '1.4'
깃 저장소가 .gemspec 파일을 가지고있지 않다면, 번들러가 의존성, 실행파일, C 확장 설정이 없는 간단한 .gemspec 파일을 생성합니다. 아마도 간단한 젬에서는 동작 할테지만, 동작하지 않을 수도 있습니다. .gemspec 파일이 없다면 깃에서 젬을 사용하지 않는 것이 좋습니다.
자세한 내용은: 깃
파일시스템에서 직접 젬을 언팩하고 싶다면, :path 옵션에 젬의 파일들이 들어있는 경로를 지정하면 간단합니다. 
gem 'extracted_library', :path => './vendor/extracted_library'
의존성들은 그룹안에 있을 수 있습니다. 그룹은 설치시에 무시(--without을 사용)하거나 한번에 require(Bundler.require를 사용)할 수 있습니다. 
gem 'wirble', :group => :development
gem 'debugger', :group => [:development, :test]

group :test do
  gem 'rspec'
end
자세한 내용은: 그룹
Gemfile에 ruby를 사용하여 필요한 루비 버전을 지정할 수 있습니다. Gemfile이 다른 루비 버전에서 로드 되었다면, 번들러는 설명과 함께 예외를 발생시킬 것 입니다. 
ruby '1.9.3'
이것이 의미하는것은 1.9.3과 ABI호환이 있는 루비VM에 의존성이 있다는 것입니다. 버전을 채크해 맞지 않으면, 번들러는 예외를 발생시킵니다. 이 것은 실행중의 코드가 매치하는지 확인합니다. :engine과 :engine_version버전 옵션을 사용해 좀 더 자세한 설정을 할 수 있습니다. 
ruby '1.9.3', :engine => 'jruby', :engine_version => '1.6.7'
