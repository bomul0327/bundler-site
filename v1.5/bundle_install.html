bundle install

어플리케이션에서 Gemfile안에 있는 의존성이 만족하는지 확인합니다. 
$ bundle install [--binstubs=PATH] [--clean] [--deployment] [--frozen]
                 [--full-index] [--gemfile=FILE] [--local] [--no-cache]
                 [--no-prune] [--path=PATH] [--quiet] [--shebang=STRING]
                 [--standalone=ARRAY] [--system] [--without=GROUP GROUP]

옵션들:

--binstubs: ./bin폴더에 실행파일의 스텁을 생성합니다.

--clean: 설치후 자동으로 bundle clean을 실행합니다.

--deployment: 배포 환경에 촤적화된 기본값으로 인스톨 합니다.

--frozen: 이번 설치 이후 Gemfile.lock을 업데이트 못하게합니다.

--full-index: API 엔드포인트 대신 rubygems의 모던 인덱스를 사용합니다.

--no-cache: 갱신에 케쉬된 젬을 사용하지 않습니다.

--no-prune: 케쉬에서 오래된 젬을 제거하지 않습니다.

--path: 시스템 기본값($BUNDLE_PATH 나 $GEM_HOME) 대신 다른 패스를 지정합니다. 번들러는 이 머신에서 이 값을 기억할 것 입니다.

--quiet: 경고와 에러만 출력합니다.

--shebang: 시스템 기본값(보통 'ruby')대신 다른 쉬뱅(#!) 실행 이름을 지정합니다.

--standalone: 번들러 런타임없이 실행되는 번들을 만듭니다.

--system: 시스템 로케일($BUNDLE_PATH 나 $GEM_HOME)에 설치합니다. 이전에 이 어플리케이션에서 어디에 설치했던 관계없습니다.

--without: 지정된 그룹의 젬들을 제외합니다.

--jobs: 병렬 워커를 사용해 인스톨합니다.

--retry: 실패한 네트워크, git 리퀘스트를 재시도합니다.

젬들은 기본 시스템 로케일에 설치될 것 입니다. 시스템 젬이 루트 권한의 장소(맥OSX 처럼)에 설치 된다면 번들은 설치를 위해 루트 패스워드를 물어볼 것입니다.

젬들을 설치할 동안, 번들러는 vendor/cache와 시스템 젬들을 체크합니다. 젬이 케쉬되거나 인스톨 되지 않았다면, 번들러는 Gemfile에서 선언한 소스에서 설치할 것입니다.

--system 옵션은 기본치입니다. 아래에 설명된 대로 --path 옵션을 사용한 이후에 기본치로 돌릴고 싶을때 넘기세요.

이미 시스템 젬에 설치되어 있어도 시스템 젬의 저장소가 아닌곳에 의존성들을 설치합니다. 이 경우에는, vendor/bundle에 설치합니다. 
$ bundle install --path vendor/bundle
또한 bundle 명령어, Bundler.setup, Bundler.require를 위해 이 위치를 기억해둡니다.
자세한 내용은: Bundler.setup 자세한 내용은: Bundler.require
명시적으로 재외한 그룹을 재외한 모든 의존성들을 설치합니다. 
$ bundle install --without development test
자세한 내용은: 그룹
프로덕션 서버에 모든 의존성들을 설치합니다. 개발머신에 이 플래그를 사용하지 마세요. 
$ bundle install --deployment

--deployment 플래그는 여러가지의 배포친화적인 규칙을 활성화시킵니다.

  • 모든 젬들을 vendor/bundle에 격리합니다.
  • 최신 Gemfile.lock을 필요로 합니다.
  • bundle package 를 실행했었다면, rubygems.org에서 가져오지 않고, vendor/cache안의 젬들만 사용합니다.
자세한 내용은: 배포하기
지정된 수의 워커를 시작하여 병행설치 합니다. 
$ bundle install --jobs 4
실패한 네트워크, git 리퀘스트를 재시도합니다. 
$ bundle install --retry 3
